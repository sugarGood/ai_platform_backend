# AI 能力平台后端总体设计

## 1. 文档定位

本文不是产品介绍，而是给后端研发、前端联调、数据库设计和网关实现直接对照开发使用的总览文档。

本文回答四个问题：

1. 原型里每个页面背后的后端模块是什么。
2. 平台的主链路到底是什么，哪些能力是核心主链路，哪些是扩展域。
3. 服务之间如何分层，哪些能力放控制面，哪些能力放运行面。
4. 本轮目标 SQL 和接口设计的边界在哪里。

## 2. 平台核心定位

AI 能力平台不是单一聊天服务，而是企业级 AI 能力治理中台，核心职责有五类：

1. 统一凭证接入：成员不直接持有上游厂商 Key。
2. 统一能力注入：知识库、技能、工具、System Prompt、MCP 集成按权限注入。
3. 统一项目上下文：同一成员可跨项目工作，但 AI 请求始终绑定当前项目上下文。
4. 统一配额与审计：所有 AI 请求、工具调用、知识检索、敏感操作可追踪。
5. 统一资产沉淀：全局能力与项目能力均可沉淀、复用、继承、订阅。

## 4. 后端服务边界

## 4.1 控制面服务

负责配置、治理、资产和权限，不直接承载模型推理流量。

主要职责：

1. 用户、部门、平台角色、项目成员和角色模板管理。
2. 平台凭证签发、轮换、吊销、客户端绑定。
3. 全局/项目知识库、技能、工具、集成、原子能力、模板管理。
4. 上游供应商、模型、路由策略、全局配额、项目配额策略管理。
5. 审计日志、活动日志、告警规则、安全规则管理。

## 4.2 运行面服务

负责承接 AI 请求和执行时上下文装配。

主要职责：

1. 校验平台凭证。
2. 判断当前项目上下文。
3. 按权限装配知识库、技能、工具、System Prompt、集成。
4. 调度上游模型并记录用量。
5. 执行知识检索、工具调用、MCP 能力代理。
6. 执行项目共享池和成员个人池双重配额检查。

## 4.3 知识处理服务

独立负责：

1. 文档解析。
2. 分块与向量化。
3. 向量库存储。
4. 检索与重排。

## 4.4 MCP 网关

对外暴露三类能力：

1. `prompts`：技能。
2. `tools`：工具。
3. `resources`：知识库资源。

同时支持：

1. 全局能力入口。
2. 项目能力入口。
3. 第三方 MCP 代理入口。

## 5. 研发必须统一的业务规则

## 5.1 一人一证，多项目复用

原型明确个人工作台只展示一张“我的平台凭证”，因此目标模型采用：

1. 成员默认只持有一张个人凭证。
2. 个人凭证可访问的项目范围来自成员项目关系和显式授权范围。
3. 服务账号凭证、临时凭证允许项目级绑定。
4. “当前工作项目”决定 AI 请求默认注入的项目上下文。

## 5.2 权限模型不是单层角色

项目权限不能只靠 `project_members.role_code` 解决，必须分三层：

1. 角色模板：Admin / Developer / QA / PM / Guest。
2. 模块权限：知识库、技能、工具、集成、成员、配额、项目设置等的访问级别。
3. 个体覆写：成员级别的权限调整和资源授权范围。

## 5.3 AI 能力配置固定为五类资产

项目 AI 能力页已经把配置面板定死为五块：

1. 知识库
2. 技能
3. 工具
4. System Prompt
5. 集成

所有数据模型、接口、菜单、权限，都必须围绕这五类资产设计。

## 5.4 配额必须采用双池模型

原型中的“项目 Token 池”和“成员个人配额”同时存在，因此每次请求都必须执行：

1. 项目共享池检查。
2. 成员个人配额检查。
3. 通过后同时扣减。
4. 任一失败都返回明确的超限原因，并写用量事件。

## 5.5 继承关系是一级核心能力

全局能力不能只是“可见”，而要支持：

1. 全局知识库被项目继承并配置检索权重。
2. 全局技能被项目订阅或克隆。
3. 平台工具被项目启用或禁用。
4. 集成市场目录与项目实际接入解耦。

## 6. 目标数据域

按开发优先级，目标数据域分为六类：

1. 平台身份与接入
2. 项目治理与权限
3. AI 资源治理
4. AI 能力资产
5. 运行与审计
6. Agent 与工作流

其中前五类是当前必须落地的主链路，最后一类可按阶段推进。

## 7. 当前 SQL 的主要问题

结合原型和现有 `docs/db/ai_platform.sql`，当前历史 SQL 有四类问题：

### 7.1 命名不统一

例如：

1. `tool_definitions` 与文档里的 `tools` 不一致。
2. `roles` 混合了平台角色和项目角色模板语义。
3. `provider_failover_policies` 只能表达主备切换，无法完整表达模型路由策略。

### 7.2 原型明确存在但 SQL 语义不完整

例如：

1. 项目角色模板缺少成员能力资源授权范围。
2. 集成市场目录和项目授权态没有彻底拆开。
3. 当前工作项目、项目端点接入和成员接入状态没有形成完整建模。

### 7.3 历史扩展域与核心主链路混杂

现有 SQL 中包含 CI/CD、DORA、Agile 等表，这些不是当前 AI 能力平台最先要打通的主链路，容易干扰建模判断。

### 7.4 约束表达不够强

例如：

1. 平台角色与项目角色的层次不清。
2. 个人凭证与服务账号的项目范围约束不够明确。
3. 项目共享池与成员个人池在命名和责任上需要明确分层。

## 8. 本轮重构后的目标

本轮文档和 SQL 调整后，研发可按以下顺序落地：

1. 先打通凭证、项目上下文、双池配额和基础网关。
2. 再落知识库、技能、工具、集成四类能力资产。
3. 最后补原子能力、工作流、审计安全和扩展治理能力。

## 9. 与实现相关的非功能要求

1. 所有明文上游 Key 仅允许存密文，不允许回显。
2. 所有 AI 请求和工具调用必须具备 `request_id` 级别追踪能力。
3. 所有敏感能力调用必须可审计，可回放参数摘要，不直接存敏感明文。
4. 所有项目能力装配必须支持缓存，但缓存命中不能绕过权限校验。
5. 所有页面级写操作都要写 `activity_logs`，敏感操作同时写 `audit_logs`。
